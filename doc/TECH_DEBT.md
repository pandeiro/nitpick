# Technical Debt Investigation

The following table lists identified sources of technical debt in the **Nitpick** codebase, along with an estimate of the effort required to address each item and the expected benefit of doing so.

| # | Area (file / construct) | Why it is debt | Suggested fix (high‑level) | Approx. effort* | Expected benefit |
|---|--------------------------|----------------|--------------------------|----------------|-------------------|
| 1 | `src/apiutils.nim` – line 109  <br> `# TODO: this is a temporary simple implementation` | Placeholder logic that likely lacks proper error handling / edge‑case coverage. | Replace with a full implementation (e.g. proper pagination, rate‑limit handling) and add unit tests. | **2 – 3 days** (write code + tests) | **High** – eliminates hidden bugs, makes API layer reliable and testable. |
| 2 | `src/auth.nim` – line 159  <br> `# TODO: This isn’t sufficient, but it works for now` | Authentication shortcut that may allow insecure or incomplete flows. | Implement the full auth flow (OAuth, token refresh, proper validation) and add comprehensive tests. | **3 – 5 days** (depends on external provider) | **High** – security & correctness; prevents future auth‑related regressions. |
| 3 | Hard‑coded “default” list name scattered across many files (`views/*`, `routes/*`, `feed.nim`, `redis_cache.nim`). | Repetition makes refactoring risky; any change to the default identifier requires edits in many places. | Introduce a `const DefaultListName = "default"` in a shared module (e.g. `src/consts.nim`) and replace all literals. | **1 day** (search‑replace + rebuild) | **Medium** – centralises a magic string, reduces future bugs when naming changes. |
| 4 | Magic numbers in `src/feed.nim` – `ChunkSize = 15`, `TotalSampleSize = 30`. | Values are opaque; tuning them requires code change and recompilation. | Expose them as configurable env vars or CLI flags, with sensible defaults. | **1 – 2 days** (add config handling + tests) | **Medium** – makes performance tuning easier without code edits. |
| 5 | Duplicate sampling logic in `fetchFeed` (blocks 49‑56 and 81‑91). | Two almost‑identical code paths increase maintenance burden and risk of diverging behaviour. | Extract the sampling routine into a private helper `proc sampleUsers(...)` and call it from both places. | **½ day** (refactor + test) | **Medium** – cleaner code, single source of truth, easier future changes. |
| 6 | Very large `fetchFeed` (≈ 190 LOC, many branches). | Hard to read, test, and reason about; hidden bugs can hide in the many branches. | Split into smaller sub‑procedures (e.g. `initSearchPool`, `buildFutures`, `processResults`). Add unit tests for each sub‑proc. | **2 days** (refactor + tests) | **High** – improves readability, testability, and future extensibility (e.g. new strategies). |
| 7 | Missing test coverage for several core modules (e.g. `auth.nim`, `apiutils.nim`, `config.nim`, parts of `redis_cache.nim`). | Bugs can slip through CI; the only safety net are the existing feed‑related tests. | Write unit tests exercising the public API of these modules (mocking Redis where needed). | **3 days** (write tests, add mock helpers) | **High** – catches regressions early, gives confidence when refactoring. |
| 8 | String literals for strategy selection (`"Sampling"`, `"Sequential"`). | Easy to typo; no compile‑time safety; scattered comparisons. | Define an `enum FeedStrategy { Sampling, Sequential }` and use it everywhere. | **½ day** (enum + conversion helpers) | **Low‑Medium** – prevents runtime mismatches, slight readability gain. |
| 9 | Global side‑effect `randomize()` at module import (`src/feed.nim`). | Makes test runs nondeterministic unless reset; can affect reproducibility. | Move randomness seeding to the program entry point (`src/nitter.nim`) or make it optional via a config flag. | **½ day** | **Low** – improves deterministic testing, negligible runtime impact. |
|10| Redis key construction logic spread across `redis_cache.nim` (multiple `if name == "default"` branches). | Repetition increases chance of typo and makes key changes cumbersome. | Centralise key generation in a small helper module (`src/redis_keys.nim`) exposing functions like `listKey(name)` etc. | **1 day** | **Medium** – easier future key‑scheme changes, reduces bugs. |
|11| Sparse documentation/comments on public procedures (many `proc`s lack docstrings). | New contributors must read source to understand intent; slows onboarding and increases risk of misuse. | Add short `##` doc comments to all exported procedures (following Nim style). | **1 day** (mostly copy‑paste) | **Low‑Medium** – improves maintainability and onboarding. |
|12| Potential race condition: `updatedPool[poolIdx] = …` while other futures are still pending (shared mutable `searchPool`). | If a future fails, the pool may end up partially updated; no explicit error handling per‑future. | Capture each pool entry in a separate immutable struct, combine results after `await allFutures`, and handle failures gracefully (retry or fail fast). | **2 days** (refactor + tests) | **Medium‑High** – makes feed fetching resilient to partial API failures. |

*Effort estimates assume an experienced Nim developer familiar with the codebase, working full‑time, and include writing/adjusting tests where relevant.

**Prioritisation (quick‑wins → high‑impact)**
1. Address the explicit TODOs (items 1 & 2) – they are known incomplete pieces and removing them eliminates the most obvious sources of bugs.
2. Add missing test coverage (item 7) – gives a safety net for any subsequent refactors.
3. Refactor `fetchFeed` (items 5, 6, 12) – large, complex function; breaking it up will make the next items easier.
4. Centralise magic strings & constants (items 3, 4, 8, 9, 10) – low effort, improves maintainability.
5. Improve documentation (item 11) – straightforward but valuable for future contributors.

Implementing these changes will markedly raise code reliability, testability, and developer velocity while keeping the effort manageable.
